## 워크로그 작성 원칙

### 핵심: "결정사항"과 "이유" 중심
- **무엇을 했는지**보다 **왜 그렇게 결정했는지** 집중
- 특히 사용자가 피드백한 내용은 반드시 포함
- 형식보다 **의미 있는 정보**가 우선

### 코드 포함 기준
- 코드는 **설명에 반드시 필요할 때만** 포함
- 단순 변경사항 나열 금지
- 핵심 결정을 보여주는 코드 스니펫만 (3-5줄 이내 권장)

### 간결함 유지
- 쓸데없는 서론, 긴 설명 배제
- 요점만 간단명료하게
- 작업 규모에 맞게 조절 (작은 작업 = 짧게, 복잡한 작업 = 상세히)

## 템플릿 (매우 유연하게 사용)

아래 구조는 참고용일 뿐, 상황에 맞게 자유롭게 변경하세요:

```markdown
# [작업명]

## 작업 개요
간단한 목적 한 줄

## 핵심 결정사항

### 1. [결정 제목]
**이유**:
- 왜 이렇게 했는지
- 다른 방법 대신 이걸 선택한 이유

**구현** (필요시):
- 핵심 변경사항만 간단히
- 코드는 꼭 필요할 때만

### 2. [결정 제목]
...

## 최종 구조 (선택)
파일 구조가 중요한 경우만

## 검증 결과 (선택)
테스트나 검증이 중요한 경우만

## 참고
- 관련 문서 링크
```

### 예시 참고
`worklogs/playwright-e2e-testing.md` 파일을 참고 - 간결하고 결정사항 중심

### 기존 파일 업데이트 방식
- 내용이 완전히 달라졌으면 **전체를 갈아쓰기** (날짜 추가 X)
- 추가 작업이면 새 섹션 추가
- 판단 기준: "나중에 읽을 때 이해하기 쉬운가?"

### 정보 정확성 관리
- 워크로그는 프로젝트의 히스토리와 컨텍스트 파악용으로 활용
- 기술적 구현 방법은 반드시 최신 공식 문서와 대조 확인
- 워크로그의 내용이 최신 공식 문서와 다른 경우 공식 문서 우선
