# 로드맵

## 목표
실생활의 모든 작업을 대신 기억해주고 사용자에게 보고한다.
핵심 아이디어는 작업을 보관하고 조회 시 제공해주는 것, 등록/상태 변경의 액션으로 사용자와 소통한다.
작업의 확장과 사용성은 각각 별도의 우선순위 트랙으로 가져간다.

### 확장성
확장을 고려한 구조를 사용해서 더 다양한 종류의 작업과 매핑시킨다.

생각해볼 만한 시나리오
- 맛집을 카테고리별로 묶어서 볼 수 있다.
- 여러 지도앱에서 맛집을 추가할 수 있다.
- 맛집을 지도뷰로 볼 수 있다.
- 구글 캘린더에서 이벤트를 볼 수 있다.
- 이벤트 시간이 다가오면 앱 알림을 받는다.
- 준비물 프리셋을 만들 수 있다.

### 사용성
극한의 편의성을 추구한다.

기대할 수 있는 가치
- 스와이프로 작업의 상태를 바꿀 수 있다.
- 음성으로 작업을 등록할 수 있다.
- 사용자 간에 리스트를 공유할 수 있다.
- 공동작업자와 작업 리스트를 함께 수정할 수 있다.
- 공동작업자와 변경사항을 실시간으로 공유할 수 있다.
- 작업을 검색할 수 있다.
- 관련 있는 작업들의 상태를 한 번에 바꿀 수 있다.
- 클립보드에서 내용을 가져와서 draft 작업으로 만들어준다.
- 중요한 작업을 위젯으로 보여준다.

## 시스템 설계
작업의 본질은 등록, 조회이기 때문에 이 성격을 띌 수 있는 모든 것을 도메인으로 엮을 수 있다.

### 작업 모델
가장 기본적인 형태는 다음과 같다:
- 제목
- 상태
- 목표일

고민이 필요한 부분
- 버전 관리를 해야 할 수 있을 것 같다
  - 다중 사용자 용으로 앱이 진화하는 경우 이력 관리가 중요해진다
- 계층이 필요할 수 있을 것 같다
  - 관련 있는 작업들을 한 번에 완료하고 싶은 경우, 일관적인 상태 전파 구조를 위해 트리 구조가 적합할 것 같다
  - 관련 있는 작업들을 묶어서 관리할 수 있다
- 태그가 필요할 수 있을 것 같다
  - 통계 같이 복잡한 형태의 조회가 필요해진다면 더 다양한 인덱스가 필요하다.
  - 관련 있는 작업들을 묶어서 관리할 수 있다
- draft 형태가 필요할 수 있을 것 같다
  - 생성의 사용성이 개선될 때 더 단순한 형태로 일단 등록할 수 있어야 한다
- 참조가 필요할 수 있을 것 같다
- 작성자가 필요할 수 있을 것 같다
  - 다중 사용자 용으로 앱이 진화하는 경우 소유, 책임, 평가 등의 개념이 필요하다

## 화면 설계
앱에서 제공할 수 있는 최선의 형태로 생성, 상태 변경 UX를 만든다.
조회의 형태는 매우 다양할 수 있다. 모두 인앱으로 풀지 말고 써드파티와의 통합이 더 중요할 수 있다.

작업을 어떻게 바라보냐에 따라서 완전히 다른 목적으로 관리하게 된다.
하단에 탭을 만들어서 자율적으로 조정할 수 있게 한다.
시스템적으로 확장적인 철학을 사용자에게도 노출한다.

# 아이디어

## 팀 협업

### 기능
- 회원가입/로그인
- 팀 생성 및 초대
- 할일 담당자 배정
- 댓글

### 기술 검토
- Docker/PostgreSQL (다중 사용자)
- Auth Service 분리
- API Gateway

## 실시간 알림

### 기능
- 브라우저 실시간 알림 (배정, 댓글, 마감)
- 이메일 알림
- 일일/주간 요약
- 알림 설정

- Kafka (이벤트 발행/구독)
- Redis (WebSocket 세션, 카운트 캐싱)
- WebSocket
- Email Service
