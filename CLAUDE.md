# Claude Code 작업 가이드

## 🎯 프로젝트 개요
- **플랫폼**: iOS 모바일 앱 (iOS 14.0+)
- **목적**: Task 관리 애플리케이션
- **Frontend**: React + TypeScript (Capacitor로 iOS 래핑)
- **Backend**: Go + Gin + GORM
- **테스팅**: Playwright (E2E), Vitest (Unit)
- **CI/CD**: GitHub Actions
- **네이티브 기능**: Push Notifications, Camera, Geolocation, Keyboard
- **상세 정보**: 각 README 참조
  - 전체 구조: `README.md`
  - Frontend: `frontend/README.md`
  - Backend: `backend/README.md`

## 1. 워크로그 참조 및 활용
- **작업 시작 전 워크로그 검토**: 관련 작업이 이전에 수행된 적이 있는지 `worklogs/` 디렉토리 확인
- **컨텍스트 파악**: 기존 워크로그에서 유사한 작업의 접근 방법, 문제점, 해결책 참조
- **일관성 유지**: 이전 작업의 패턴과 결정사항을 고려하여 일관된 접근 방식 적용
- **경험 활용**: 과거의 실패 사례나 성공 사례를 바탕으로 더 효율적인 작업 진행
- **작업 완료 후**: 새로운 워크로그 생성 또는 기존 워크로그 업데이트로 지식 축적

## 2. Plan Mode 작성 가이드

### 2.1 계획서 깊이 결정
계획의 복잡도는 작업의 영향 범위에 따라 조정:

**심층 분석이 필요한 경우:**
- 여러 파일/모듈에 걸친 구조적 변경
- 기존 패턴의 마이그레이션 또는 리팩토링
- 여러 구현 방식 중 선택이 필요한 경우
- 플랫폼/환경별로 다른 접근이 필요한 경우
- 아키텍처 결정이나 기술 스택 변경

**간단한 계획으로 충분한 경우:**
- 단일 파일/함수의 변경
- 명확한 요구사항의 기능 추가
- 문서화 작업
- 명백한 버그 수정

### 2.2 심층 계획서 구조

복잡한 작업의 경우 다음 구조를 권장:

#### 1) Current State (현재 상태 분석)
- 관련된 모든 구현체/파일 조사
- 각 구현의 상태를 시각적으로 표시
  - ✅ 이미 요구사항 충족
  - ❌ 변경 필요
  - ⚠️ 부분적으로 문제 있음
- 코드 예시와 위치 명시 (file_path:line_number)

#### 2) Problem (문제 정의)
- 왜 변경이 필요한가
- 현재 접근 방식의 한계
- 기술적 제약사항
- 해결하지 않을 경우의 영향

#### 3) Solution Plan (해결 방안)
- 여러 옵션이 있다면 모두 나열
- 각 옵션의 장단점 비교
- 선택한 방안과 그 이유
- 단계별 접근 방법
- 예상되는 리스크와 대응 방안

#### 4) Files to Modify (변경 대상)
- 수정할 파일 목록
- 각 파일에서 수행할 구체적 작업
- 파일 간 의존성이 있다면 순서 명시
- 테스트 파일이나 문서 업데이트도 포함

## 3. 개발 및 테스트 방법

### 3.1 개발 서버 실행

```bash
# 프로젝트 루트에서 실행 (프론트엔드 + 백엔드 동시 실행)
make dev

# 브라우저: http://localhost:5173
# 백엔드: http://localhost:8080
```

⚠️ **중요**: 프론트엔드와 백엔드는 상호 의존적이므로 반드시 `make dev`로 함께 실행

### 3.2 테스트 방법

**백엔드 API 문제 발생 시**
- 먼저 curl로 API를 직접 호출하여 백엔드 동작 확인 권장
```bash
curl http://localhost:8080/health
curl http://localhost:8080/api/v1/tasks
```

**E2E 테스트**
```bash
# 터미널 1
make dev

# 터미널 2
make test-e2e                    # 개발 환경
make test-e2e PHASE=release      # 프로덕션 빌드
```

**상세 정보**: `README.md` 참조
